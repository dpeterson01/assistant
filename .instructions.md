# Atlas Assistant System

## Core Setup

```sh
ATLAS="python3 ~/projects/personal/assistant/scripts/atlas-db.py"
```

Things 3 scripts: `~/.local/bin/things3/`
Contacts scripts: `~/.local/bin/contacts/`

## First Steps (every session)

1. Run `date '+%A %B %d, %Y'` in a terminal to determine today's date. **Shell `date` is the single source of truth for day-of-week.** Never derive day-of-week from a date string yourself. If the invocation prompt or user message says a different day-of-week, `date` wins.
2. Read `/memories/identity.md` and `/memories/priorities.md`.
3. Read `assistant/data/context/action-items.md` and `assistant/data/context/waiting-on-others.md`.
4. Read `assistant/data/context/objectives.md` for this week's objectives and today's MITs.
5. Run `$ATLAS sync-things3` to pull Things 3 completions into the DB.

## Data Architecture

See [data-architecture.md](assistant/context/data-architecture.md) for full query/mutation reference.

- **Source of truth for task state**: Things 3. Users interact with tasks in Things 3 (phone, watch, Mac). Atlas DB (`assistant.db`) is the analytics and tracking layer that syncs from Things 3.
- **Generated views** (never edit manually): `assistant/data/context/action-items.md`, `assistant/data/context/waiting-on-others.md`, and `assistant/data/context/objectives.md`. These are rendered by `$ATLAS` commands automatically.
- **Briefing JSON** (`briefings/YYYY-MM-DD_daily_brief.json`) powers the dashboard. The markdown briefing is a disposable render of the JSON. Never write task state back to markdown.
- **Dashboard** is read-only for task state. It displays current state. Completions happen in Things 3 or via `$ATLAS commit complete`.

## Task Lifecycle

- **Create**: `$ATLAS commit add ...` (auto-generates Task ID, pushes to Things 3, re-renders markdown views)
- **Complete**: `$ATLAS commit complete --task-id AI-...` (marks done in DB + Things 3, re-renders markdown)
- **Cancel**: `$ATLAS commit cancel --task-id AI-...`
- **Nudge**: `$ATLAS commit nudge --task-id AI-... --channel email`
- **Query**: `$ATLAS commit list`, `$ATLAS commit search`, `$ATLAS commit overdue`
- **Meetings**: `$ATLAS meeting list`, `$ATLAS meeting add`, `$ATLAS meeting recap`
- **Interactions**: `$ATLAS interaction log`

## Objectives & MITs

Weekly objectives (top 3) set during Sunday review. Daily MITs (Most Important Tasks, top 3) set each morning. Stored in `assistant.db` (`objectives`, `daily_mits` tables), rendered to `context/objectives.md`.

- **Set objective**: `$ATLAS objective set --rank 1 --title "..." --context work` (status defaults to proposed)
- **Activate objective**: `$ATLAS objective set --rank 1 --title "..." --status active`
- **List objectives**: `$ATLAS objective list` (current week) or `--week 2026-W19`
- **Link task to objective**: `$ATLAS objective link --objective-id OBJ-2026W19-1 --task-id AI-...` (also tags Things 3 task)
- **Score/close objective**: `$ATLAS objective score --rank 1 --status completed --score 8`
- **Carry forward**: `$ATLAS objective carry --to-week 2026-W20`
- **Set MIT**: `$ATLAS mit set --rank 1 --title "..." --objective-id OBJ-...` (optionally `--task-id AI-...` to tag Things 3)
- **Complete MIT**: `$ATLAS mit complete --rank 1`
- **List MITs**: `$ATLAS mit list` (today) or `--date YYYY-MM-DD`
- **MIT scorecard**: `$ATLAS mit score`

## Execution Rules

- **Resilience**: If a tool call fails, retry ONCE. If it fails again, log "⚠️ [tool/step] failed: [reason]", skip, continue. Never block on a single source. Report skipped items at the end.
- **Parallelism**: Fire independent data sources simultaneously (email, Things 3, WorkIQ, contacts are all independent). Only sequence calls that depend on prior results.
- **Terminal timeouts**: 10s for quick commands (Things 3 scripts, ls, file reads). 30s for longer operations (email cleanup, batch scripts). Never use timeout=0.
- **Progress**: If a step takes multiple tool calls without progress, skip with a note and move on.

## Hard Rules

- Never manually edit `action-items.md`, `waiting-on-others.md`, or `objectives.md`. They are generated views.
- Never create tasks for GitHub pull requests or code reviews. Surface PR review requests as informational items only.
- Briefing markdown is a rendered view of the JSON. Never write back to it.
- Access requests are hard-excluded from all briefings and triage. Filter silently.
- Apply the 5-point action item extraction test before creating any task from inbound communications.

## Gotchas

| Issue | Fix |
|-------|-----|
| Briefing shows wrong day-of-week | Always run `date '+%A %B %d, %Y'` as first step. Never derive day-of-week from date strings. |
| Things 3 `search.sh` matches wrong task (substring) | Use `$ATLAS commit complete --task-id AI-...` for exact matching. Keyword search is a fallback. |
| Terminal command hangs | Always set explicit timeouts. Never use timeout=0. |
| WorkIQ returns partial/empty data | Retry ONCE. If still empty, proceed and note "⚠️ WorkIQ unavailable". |
| Triage creates tasks for FYI emails | Apply the 5-point action item extraction test before creating any task. |
| atlas-db `commit add` fails with "task_id already exists" | Item already tracked. Use `$ATLAS commit search` to find it. |
| Completed item reappears in carryOver | Always filter carryOver against `$ATLAS commit list --status completed`. |
| Work journal includes personal data | Hard boundary. Never include iMessage, personal email, or non-Microsoft data in work journals. |
| Contact lookup fails on name | Read `index.json` first. Match on name, aliases, or email. |
